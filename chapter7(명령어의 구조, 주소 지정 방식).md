1. 명령어의 기본 구성 요소: 연산 코드와 오퍼랜드
명령어는 기본적으로 "무엇을 대상으로(오퍼랜드), 무엇을 수행하라(연산 코드)"는 형식을 따릅니다. 이는 일상적인 명령 구조와 일맥상통합니다.
인간의 명령 예시: "학생들(대상), 과제를 제출하세요(연산)", "강아지야(대상), 이것을 물어와(연산)"
컴퓨터 명령어 구조: 연산 코드(Opcode) + 오퍼랜드(Operand)
연산 코드(Opcode)와 오퍼랜드(Operand)의 정의
연산 코드 (Operation Code): 명령어가 수행할 구체적인 동작(더하기, 저장, 호출 등)을 정의합니다.
오퍼랜드 (Operand): 연산의 대상이 되는 데이터 자체 혹은 그 데이터가 저장된 **'위치(주소)'**를 명시합니다.
오퍼랜드 필드가 '주소 필드'로 불리는 이유
오퍼랜드 필드에는 실제 데이터가 담기기도 하지만, 하드웨어의 물리적 비트 제한으로 인해 데이터가 저장된 메모리 주소나 레지스터 이름이 담기는 경우가 훨씬 많습니다. 이러한 특성 때문에 오퍼랜드 필드를 흔히 주소 필드라고 부르기도 합니다.
오퍼랜드 개수에 따른 명령어 분류
명령어 내 오퍼랜드의 개수는 고정되어 있지 않으며, 연산의 성격에 따라 유동적입니다.
0-주소 명령어: 오퍼랜드가 없음 (예: HALT(종료), RETURN).
1-주소 명령어: 오퍼랜드가 1개.
2-주소 명령어: 오퍼랜드가 2개.
3-주소 명령어: 오퍼랜드가 3개.

2. 연산 코드(Opcode)의 4대 유형 및 자료구조의 활용
CPU 아키텍처마다 고유한 연산 코드를 설계하지만, 범용적으로는 다음 네 가지 핵심 유형으로 분류됩니다.
1) 데이터 전송 (Data Transfer)
데이터의 위치를 옮기는 연산입니다.
LOAD (FETCH): 메모리의 데이터를 CPU(레지스터)로 가져오는 동작.
STORE: CPU의 데이터를 메모리에 저장하는 동작.
PUSH / POP: 스택(Stack) 자료구조를 활용한 데이터 입출력.

큐 (Queue)

2) 산술/논리 연산 (Arithmetic/Logic)
사칙연산(ADD, SUB, MUL, DIV)과 논리 연산(AND, OR, NOT), 그리고 값의 크기를 비교하는 연산을 포함합니다.
3) 제어 흐름 변경 (Control Flow)
프로그램의 실행 순서를 강제로 바꾸는 연산입니다.
JUMP / CONDITIONAL JUMP: 특정 주소로 무조건 혹은 조건부로 건너뜁니다.
CALL / RETURN: 함수를 호출할 때 사용됩니다. 특히 CALL은 나중에 돌아올 '복귀 주소(Return Address)'를 미리 저장한 뒤 특정 주소로 이동한다는 점에서 단순 점프와 구별됩니다.
4) 입출력 제어 (I/O Control)
입출력 장치와 데이터를 주고받거나 장치의 상태를 테스트하는 연산입니다.

3. 오퍼랜드 지정과 유효 주소(Effective Address)의 필요성
명령어 내에 데이터를 직접 넣지 않고 '주소'를 사용하는 결정적 이유는 명령어의 비트(Bit) 수 제한이라는 물리적 한계에 있습니다.
비트 제한에 따른 수치 비교 분석 (16비트 명령어 예시)
전체 명령어 크기가 16비트이고, 연산 코드가 4비트를 차지한다고 가정해봅시다.
데이터 직접 포함 (3-주소 명령어): 남은 12비트를 3개의 오퍼랜드로 나누면 하나당 4비트씩 할당됩니다. 이 경우 표현 가능한 데이터 값은 2 
4
 =16개에 불과하여 큰 수의 연산이 불가능합니다.
주소 명시 방식: 오퍼랜드 필드에 메모리 주소를 넣으면, 해당 주소(위치)에 저장된 16비트 혹은 그 이상의 방대한 데이터를 그대로 활용할 수 있습니다. 예를 들어 16비트 메모리 주소 체계를 참조한다면 2

유효 주소 (Effective Address)
연산에 사용될 데이터가 최종적으로 저장된 실제 위치(메모리 주소 또는 레지스터)를 유효 주소라고 정의합니다.

4. 명령어 주소 지정 방식(Addressing Modes) 심층 분석
CPU가 유효 주소를 결정하는 방식은 시스템의 성능과 메모리 활용 범위 사이의 트레이드오프를 결정합니다. 핵심적인 성능 지표는 메모리 접근 횟수입니다.

성능 분석: 레지스터는 CPU 내부에 존재하고, 메모리는 CPU 외부에 물리적으로 떨어져 존재합니다. 따라서 레지스터를 활용하는 방식이 물리적 거리와 버스 속도 차이로 인해 훨씬 효율적입니다.
